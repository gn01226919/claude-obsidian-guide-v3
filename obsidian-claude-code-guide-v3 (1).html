核心思維

Claude Code 很強，但它有一個根本限制：每次開新 session 都是空白的，完全不記得上次的對話。你的思考脈絡、做過的決定、踩過的坑，全部消失。Obsidian Vault 就是用來解決這個問題——它是 Claude 的「外部硬碟」，讓知識可以跨 session 保存。

01 — 問題背景

沒有 Vault 之前，你在浪費什麼？

這些問題你可能每天都在遇到，但已經習慣了，不覺得是問題。

❌ 現在的狀況
  • 每次開新 session 要重新解釋「我在做什麼」
  • 做了一個技術決定，三個月後忘記原因，又重新想一遍
  • 多個專案同時進行，上下文容易混亂
  • 靈感一來沒地方記，想記又懶得整理
  • 踩過的坑沒有記錄，之後再踩一次
  • GitHub 有 code，但沒有「為什麼這樣寫」
✅ 有 Vault 之後
  • 叫 Claude 讀 Vault，直接知道所有背景
  • 所有決定都有紀錄：決定了什麼、為什麼、當時的條件
  • 每個專案獨立資料夾，切換時脈絡清晰
  • 靈感丟進 inbox，叫 Claude 幫你整理
  • bugs-log.md 記住所有踩過的坑
  • Code + 脈絡 = 完整的專案記憶

為什麼這個問題比你想的更嚴重？

假設你做一個 SaaS 產品，從開始到 MVP 要三個月。這三個月你和 Claude Code 至少有 100 次對話。每次對話平均花 10 分鐘重新交代背景，就是 16 個小時浪費在解釋你自己已經知道的事情上。

更嚴重的是決策品質的損失：沒有決策紀錄，你會在同一個問題上反覆猶豫，因為你不記得自己當初為什麼做這個選擇。有了 Vault，你可以叫 Claude「去看我之前對這個問題的分析，幫我評估現在的狀況有沒有改變」，直接站在過去的思考基礎上繼續。

02 — Obsidian 基礎

Obsidian 到底是什麼？

很多人聽到「知識管理工具」就覺得複雜，其實 Obsidian 的本質很簡單。

最簡單的理解方式

你現在電腦上有很多 .md 檔案，比如專案的 README.md、Claude Code 用的 CLAUDE.md。你平時用 VS Code 或任何文字編輯器打開這些檔案。

Obsidian 就是一個專門用來讀寫 .md 檔案的介面，但它額外提供了幾個功能：筆記之間可以互相連結、有視覺化的連結圖、有強大的全文搜尋。就這樣。

你的資料不在 Obsidian 的伺服器，不在雲端，就在你電腦的某個資料夾裡。你關掉 Obsidian，用 VS Code 打開同一個資料夾，所有內容都還在，格式完全一樣。

ℹ️
Vault 是什麼？

「Vault」是 Obsidian 對「工作資料夾」的稱呼。你告訴 Obsidian「這個資料夾是我的 Vault」，它就會把那個資料夾裡所有的 .md 檔案當成你的筆記來管理。一個 Vault = 一個資料夾。你可以有多個 Vault，但通常一個就夠了。

四個你會真正用到的功能

🔗

雙向連結

在任何筆記裡輸入 [[其他筆記名稱]]，Obsidian 就會自動建立連結。被連結的筆記也會知道「有人連結我」，這叫 backlink。

範例：在「會員系統決策」裡寫 [[OTP 服務比較研究]]，點擊就能跳到那篇。

🕸️

Graph View（連結圖）

按下 Ctrl+G（Mac: Cmd+G），所有筆記和它們之間的連結會以視覺化的圖形呈現。一眼就能看到哪些概念之間有關聯，哪些筆記是孤立的。

對於管理多個同時進行的專案特別有用。

🔍

全文搜尋

Ctrl+Shift+F（Mac: Cmd+Shift+F），瞬間搜尋所有筆記的內容。比 grep 更快，也能搜標題、tags、內容。

找三個月前某個決定的理由，幾秒就找到。

⚡

快速切換

Ctrl+O（Mac: Cmd+O），打幾個字母就能跳到任何一篇筆記。不用用滑鼠慢慢找。

整個工作流可以完全用鍵盤完成。

03 — 安裝設定

從零開始的完整設定流程

每個步驟都說明你在做什麼、為什麼這樣做。預計完成時間：30 分鐘。

1

下載 Obsidian

前往 obsidian.md 下載對應你作業系統的版本。這是免費的，不需要建立帳號。安裝完直接打開。

ℹ️

第一次開啟會問你要「新建 Vault」還是「打開現有資料夾」。先選新建 Vault，繼續下一步。

2

建立 Vault 資料夾結構

打開終端機（Terminal / iTerm 或任何你習慣的終端機），執行以下指令。每行的意思都有說明。

bash 建立 Vault 基礎結構 — 逐行解釋 
# mkdir = make directory，建立資料夾
# -p = 如果父資料夾不存在，一起建立；已存在也不報錯
# ~ = 你的家目錄（通常是 /Users/你的名字/）
mkdir -p ~/vault

# cd = change directory，切換到那個資料夾
cd ~/vault

# 用一個指令同時建立多個子資料夾
# {} 裡的名稱用逗號隔開，不能有空格
mkdir -p projects inbox decisions resources daily-notes

# ls = list，列出目前資料夾裡的所有內容
# -la = 顯示詳細資訊（包含隱藏檔案）
# 執行這個來確認資料夾都建好了
ls -la

執行後你應該看到這幾個資料夾：

output預期輸出
drwxr-xr-x  decisions
drwxr-xr-x  daily-notes
drwxr-xr-x  inbox
drwxr-xr-x  projects
drwxr-xr-x  resources
3

在 Obsidian 裡開啟這個 Vault

回到 Obsidian，選擇 Open folder as vault，導航到 ~/vault（就是你家目錄下的 vault 資料夾），點 Open。

Obsidian 現在就能看到你剛才建的資料夾結構了。左側欄位會顯示所有子資料夾。

⚠️
常見誤解

有些人會想說「那我直接把 code repo 當 Vault 不就好了？」這樣做會有問題：你的 Vault 裡會混進 node_modules、.gitignore、package.json 等非筆記檔案，搜尋和連結功能會很雜亂。建議 Vault 和 code repo 分開，保持 Vault 裡只有 .md 筆記。

4

建立你的第一個專案狀態檔

這是最重要的檔案，記錄你的專案現在在哪裡、下一步要做什麼。用以下指令建立範本：

bash 建立專案狀態檔範本 
# 先建立專案子目錄
# 把 "my-project" 換成你的專案名稱
mkdir -p ~/vault/projects/my-project

# cat > 檔案路徑 << 'EOF' 是 heredoc 語法
# 讓你可以在終端機直接輸入多行內容
# 直到遇到 EOF 才結束
cat > ~/vault/projects/my-project/status.md << 'EOF'
# My Project — 狀態追蹤

最後更新：（每次工作結束更新這裡）

## ✅ 已完成
- （列出完成的功能/任務）

## 🔄 進行中
- （目前正在做的）

## ⏳ 下一步（按優先順序）
1. 
2. 
3. 

## ❓ 還未決定的問題
- （需要之後再討論的）

## ⚠️ 注意事項 & 踩過的坑
- （避免重蹈覆轍的提醒）

## 📁 相關檔案路徑
- Code repo：~/（你的專案路徑）
- 決策紀錄：[[decisions]]
EOF
5

確認 Claude Code 可以讀取 Vault

bash 啟動並測試 
# 切換到 vault 目錄（關鍵步驟）
cd ~/vault

# 啟動 Claude Code
claude

# 在 Claude Code 裡輸入以下測試指令：
# "請列出這個 vault 裡所有的 .md 檔案，並說明結構"
✅
成功的樣子

Claude Code 應該會回答類似：「我看到你的 vault 有以下結構：projects/my-project/status.md、inbox/、decisions/ 等資料夾...」代表它可以正常讀取你的 Vault。

⚠️
為什麼一定要從 vault 目錄啟動？

Claude Code 啟動時，它的「工作目錄」就是你執行 claude 指令的那個資料夾。從 vault 目錄啟動，它就能看到所有筆記。如果你從其他地方啟動，你就需要每次都告訴它 vault 的完整路徑，比較麻煩。

04 — 資料夾設計

Vault 的完整結構設計

好的結構讓你記錄更快、Claude 找脈絡更準。這裡說明每個資料夾的作用和裡面應該放什麼。

tree 完整目錄結構與說明 
~/vault/
│
├── CLAUDE.md                    # ← 最重要，Claude 每次啟動都讀這個
│
├── projects/                    # 每個進行中的專案一個子資料夾
│   ├── project-a/
│   │   ├── status.md            # 現在做到哪、下一步是什麼
│   │   ├── decisions.md         # 所有決定的紀錄（含理由）
│   │   ├── architecture.md      # 系統設計、技術選型
│   │   └── bugs-log.md          # 踩過的坑和解法
│   └── project-b/
│       └── ...
│
├── inbox/                       # 靈感暫存區，不需要整理就可以丟
│   └── quick-capture.md         # 所有快速記錄都先進這裡
│
├── decisions/                   # 跨專案的重大決定（技術棧、架構等）
│   ├── tech-stack.md
│   └── architecture-principles.md
│
├── resources/                   # 學習筆記、參考資料、技術研究
│   ├── supabase-tips.md
│   ├── nextjs-patterns.md

    ├── 2026-03-26.md
    └── 2026-03-27.md

每個資料夾的詳細說明

📁

projects/

放什麼：每個進行中的專案，各自一個子資料夾。

為什麼這樣分：讓 Claude 讀取時可以精準指定「只看這個專案的脈絡」，不會把不同專案的資訊混在一起。

📥

inbox/

放什麼：任何還沒整理的想法、靈感、待辦。

使用方式：先丟進來，不用考慮格式或放哪裡。之後叫 Claude 幫你整理到對應的位置。這樣降低記錄的摩擦力。

⚖️

decisions/

放什麼：跨多個專案都適用的重大決定，例如「所有專案都用 TypeScript」、「統一用 Supabase 做資料庫」。

為什麼不放在 projects/ 裡：這些決定不屬於單一專案，而是你整體的技術方向。

📚

resources/

放什麼：你學到的東西——某個函式庫的使用技巧、某個問題的解法研究、某篇文章的重點摘要。

與 inbox 的差別：inbox 是未整理的暫存，resources 是整理好的知識。

## 2026-03-15 選擇 Supabase 而非 Firebase

**決定：** 使用 Supabase

**評估過的選項：**
- Firebase：熟悉，但 NoSQL 對這個專案的關聯資料不適合
- Supabase：PostgreSQL 基礎，有 Row Level Security，費用更低
- 自建 PostgreSQL：控制性最高但維運成本太高

**選 Supabase 的理由：**
1. 關聯式資料庫符合我們的資料結構
2. 內建 Auth 省去自建認證的工作
3. 免費方案足夠 MVP 階段使用

**未來重新評估的條件：**
- 月活躍用戶超過 10,000 時檢視費用
markdown bugs-log.md — 踩過的坑，避免下次再踩 
## 2026-03-20 Supabase RLS 政策導致查詢返回空陣列

**症狀：** 查詢看起來正確，但永遠返回 []

**原因：** Row Level Security 啟用後，沒有對應的 policy
允許讀取，所以所有查詢都被擋掉了。

**解法：**
```sql
CREATE POLICY "Allow read for authenticated users"
ON your_table FOR SELECT
USING (auth.uid() IS NOT NULL);
```

**下次遇到查詢返回空時，先檢查 RLS policies。**
05 — 核心設定

CLAUDE.md：你和 AI 之間的永久契約

這是整個系統最重要的檔案。放在 vault 根目錄，Claude Code 每次從這個目錄啟動就會自動讀取。相當於你永久告訴 Claude「你是誰、在做什麼、有什麼規則」。

為什麼要有 CLAUDE.md？

正常情況下，你每次開新 session，都要重新告訴 Claude 你的背景。有了 CLAUDE.md，這些資訊會自動載入，你不用再說。

另外，CLAUDE.md 也可以放「規則」——比如「不要刪任何檔案」、「每次做完要更新 status.md」、「回答用繁體中文」。這些規則每次 session 都自動生效。

bash 建立 CLAUDE.md 
# 在 vault 根目錄建立 CLAUDE.md
cat > ~/vault/CLAUDE.md << 'EOF'
（把下面的範本內容貼進去，然後根據你的情況修改）
EOF
markdown CLAUDE.md 完整範本（根據你的情況修改每個 [] 的部分） 
# CLAUDE.md — [你的名字] 的開發 Vault

---

## 關於我
- **身份：** [你的職業，例如：獨立開發者、前端工程師]
- **主要技術棧：** [例如：Next.js, TypeScript, Supabase, Tailwind CSS]
- **溝通語言：** 繁體中文（回答請用繁體中文）
- **經驗程度：** [例如：剛開始學習、有 2 年經驗]

## 現在進行中的專案

### 專案 A：[專案名稱]
- **一句話描述：** [例如：一個給美髮沙龍用的會員管理系統]
- **技術棧：** [例如：Next.js + Supabase + Shadcn UI + Stripe]
- **目前階段：** [例如：MVP 開發中，預計 4 月上線]
- **狀態檔案：** projects/project-a/status.md

### 專案 B：[專案名稱]
- **一句話描述：** [描述]
- **狀態檔案：** projects/project-b/status.md

## 規則（每次工作都要遵守）

### 開始工作前
1. 問我要做哪個專案
2. 讀取對應的 projects/[名稱]/status.md
3. 告訴我現在的狀態，然後等我說要做什麼

### 工作中
- 做了重要決定時，問我要不要記錄到 decisions.md
- 遇到 bug 並解決後，問我要不要記錄到 bugs-log.md
- 不確定的事情先問，不要自己猜測然後亂改

### 結束工作前
- 更新對應專案的 status.md（完成了什麼、下一步是什麼）
- 提醒我有哪些事情還沒決定

### 永遠遵守
- **不刪任何檔案**，除非我明確說「請刪掉 [檔案名稱]」
- **回答用繁體中文**
- **解釋指令在幹嘛**，不要只給我指令讓我盲目執行

🎯

規則要具體

「要小心一點」這種規則沒用。「不刪任何檔案，除非我明確確認」這種具體規則才有效。規則越具體，AI 越能確實遵守。

06 — 實際情境

8 個你每天都會遇到的情境

每個情境說明問題是什麼、為什麼會發生、以及你要怎麼做（含完整指令）。

1
每天開始工作 — 接續上次的狀態

問題：你昨天做到一半，今天開新 session，完全忘了上次在幹嘛。你需要花時間回憶，或重新跟 Claude 解釋一遍。

為什麼沒有 Vault 會這麼痛：Claude 不記得昨天的對話，每次都是空白開始。如果你也沒有紀錄，就沒有人知道「上次做到哪」。

流程 每天開始工作的標準程序 
# 步驟 1：進入 vault 目錄
cd ~/vault

# 步驟 2：啟動 Claude Code
claude

# 步驟 3：在 Claude Code 裡說：
"請讀取 projects/my-project/status.md，
告訴我現在的狀態和下一步是什麼，
然後等我說要從哪裡開始。"

Claude 會回答類似：「你的專案目前完成了 A 和 B，下一步是 C。根據 status.md，上次停在 D 這個問題還沒解決。你要先處理哪個？」你立刻進入狀況，完全不需要回憶。

2
做了技術決定 — 記錄下來，未來可以查

問題：你和 Claude 討論了一個小時，選定了某個方案。但這個決定的理由都在那次對話裡，之後 session 結束，理由就消失了。

情境範例：你在考慮要用哪個第三方服務，討論了 A 方案 vs B 方案 vs C 方案，最終選了 A。三個月後有人問你為什麼選 A，你完全不記得了。

指令 決策結束後立刻執行 
"請把我們剛才關於 OTP 服務選擇的討論，
整理成一篇決策紀錄，存到 projects/my-project/decisions.md。
格式包含：今天的日期、我們的決定是什麼、
評估過的選項、選擇這個的理由、
以及在什麼條件下我們應該重新評估這個決定。"

Claude 會把整個討論過程濃縮成結構化的決策紀錄。以後要查，一秒找到。

💡

進階用法：三個月後你想重新評估某個決定，叫 Claude：「讀取 decisions.md 裡的 [決定名稱]，告訴我當時的評估條件，幫我看看現在這些條件有沒有改變。」

3
靈感突然來 — 快速記錄，不破壞工作流

問題：你在做 A 功能的時候，突然想到 B 功能應該怎麼做，或者有個很好的 UI 想法。你想記下來，但打開 Obsidian 整理又太麻煩，所以最後就忘了。

bash 最快的記錄方式（不用打開 Obsidian） 
# 直接在終端機把想法加到 inbox
# echo 輸出文字，>> 代表「附加到檔案末尾」（不是覆蓋）
# 如果檔案不存在，>> 會自動建立它
echo "登入頁考慮加 Google OAuth，減少手機 OTP 的摩擦" >> ~/vault/inbox/quick-capture.md

# 如果想加上日期標記
echo "$(date +%Y-%m-%d) 登入頁考慮加 Google OAuth" >> ~/vault/inbox/quick-capture.md

# 確認有沒有存進去
cat ~/vault/inbox/quick-capture.md

等到你有時間的時候，叫 Claude 整理 inbox：

指令 整理 inbox 的指令 
"讀取 inbox/quick-capture.md，
把裡面的內容分類整理到對應的位置：
- 跟 my-project 有關的，加到 projects/my-project/ 的對應檔案
- 技術研究的想法，加到 resources/
- 跨專案的決定，加到 decisions/
告訴我你把什麼搬到哪裡了，然後清空 quick-capture.md。"
4
踩到 Bug — 記錄解法，避免下次再踩

問題：你花了兩小時解決一個 bug，解掉了，繼續往前。兩個月後，同樣的 bug 出現在另一個地方，你又花了兩小時。

指令 解完 bug 立刻執行 
"把剛才解決的這個 bug 記錄到 projects/my-project/bugs-log.md。
格式：
- 日期
- 症狀（看到什麼錯誤訊息、什麼行為）
- 根本原因（為什麼會發生）
- 解法（完整的修復步驟或代碼）
- 之後怎麼避免（預防措施）"

下次遇到類似問題，你可以叫 Claude：

指令 查詢歷史 bug 記錄 
"讀取 projects/my-project/bugs-log.md，
看看有沒有跟我現在遇到的問題（[描述症狀]）類似的紀錄，
如果有，告訴我之前的解法。"
5
多專案切換 — 不把不同專案的脈絡混淆

問題：你同時在做三個專案。做完 A 專案切換到 B 專案時，Claude 有時候會把 A 的技術細節帶進 B 的討論裡，造成混亂。

指令 切換專案的標準流程 
"我們現在切換到 project-b。
請做以下幾件事：
1. 把剛才 project-a 的所有細節清出你的工作記憶
2. 讀取 projects/project-b/ 資料夾裡所有的 .md 檔案
3. 告訴我這個專案現在的狀態和技術棧
4. 等我說要做什麼"
ℹ️

「清出工作記憶」這個說法很重要。明確告訴 Claude 要切換脈絡，它才不會把上一個專案的資訊混進來。這比開新 session 更快，也不需要重新建立基本背景。

6
每天結束工作 — 更新進度，為明天做準備

問題：今天做了很多事，但沒有記錄。明天開始工作又要重新想「昨天做到哪了」。

解法：每次工作結束前，固定執行這個指令（養成習慣，只要 30 秒）：

指令 每天工作結束前的固定指令 
"工作結束了，請幫我更新 projects/my-project/status.md：
1. 把今天完成的功能/任務移到「已完成」清單
2. 更新「進行中」的項目
3. 把「下一步」更新成明天應該優先做的事
4. 記錄今天遇到但還沒解決的問題
5. 更新最後修改日期
完成後給我看更新後的內容，讓我確認。"

這 30 秒的習慣，讓明天的你直接進入狀況，不需要回憶。

7
查找過去的資訊 — 不靠記憶，靠搜尋

問題：「我記得我之前研究過這個問題，但忘了結論是什麼」——這種情況太常見了。

bash 在 Vault 裡搜尋關鍵字（終端機指令） 
# grep = 搜尋文字，-r = 包含子資料夾，--include = 只搜 .md 檔
# -l = 只顯示有找到的檔案名稱
# -i = 不分大小寫

# 找哪些檔案裡有提到某個關鍵字，例如 "API"
grep -r -l -i "API" ~/vault --include="*.md"

# 找所有提到該關鍵字的內容（含檔案名稱和行數）
grep -r -n -i "API" ~/vault --include="*.md"

# 在 Obsidian 裡用圖形介面搜尋
# 按 Cmd+Shift+F（Mac）或 Ctrl+Shift+F（Windows）
指令 叫 Claude 去找 
"請在整個 vault 裡搜尋跟 'OTP' 相關的所有筆記，
告訴我在哪些檔案裡有提到，以及各自說了什麼。"
8
知識沉澱 — 把學到的技術記下來，以後直接用

問題：你研究了一個技術，搞懂了，繼續往前。下次遇到類似問題，又要重新研究一遍。

解法：每次解決一個技術問題後，把學到的東西存進 resources/。

指令 把研究結果存成知識筆記 
"把我們剛才解決 Supabase RLS 問題的過程，
整理成一篇技術筆記存到 resources/supabase-tips.md。
格式：
- 問題描述（什麼情境下會遇到）
- 原因解釋（背後的機制）
- 解法步驟（可以直接複製的代碼）
- 相關參考資源
讓我下次遇到類似問題，讀這篇就能解決，不用再查文件。"

一個月後，你的 resources/ 資料夾就會有一個「你自己的技術手冊」，全部都是你真正用過、驗證過的解法。

07 — 指令總覽

所有你需要的指令，逐一解釋

這裡把所有指令集中在一起，每個都說明在幹嘛。可以當作備忘清單使用。

終端機指令（bash）

這些是你在終端機輸入的指令，用來建立結構、啟動 Claude、搜尋等。

cd ~/vault 切換工作目錄到你的 Vault
cd = change directory，切換資料夾。~ 是你的家目錄（/Users/你的名字/）。這個指令讓你「進入」vault 資料夾，之後執行的所有指令都以這裡為基準。一定要先執行這個，再啟動 Claude Code。
claude 啟動 Claude Code
在你 cd ~/vault 之後再執行這個。Claude Code 會以當前資料夾（vault）作為工作目錄啟動，就能讀取所有的 .md 筆記。
mkdir -p ~/vault/projects/new-project 建立新專案資料夾
mkdir = make directory，建立資料夾。-p = 如果中間的資料夾不存在也一起建立，且不會因為已存在而報錯。把 new-project 換成你的專案名稱。
echo "文字" >> ~/vault/inbox/quick-capture.md 快速把文字附加到 inbox 檔案末尾
echo 把引號裡的文字輸出。>> 代表「附加到檔案末尾」（不會覆蓋原有內容，如果用 > 就會覆蓋，注意差別）。如果 quick-capture.md 不存在，>> 會自動建立它。
cat ~/vault/inbox/quick-capture.md 顯示 inbox 裡的所有內容
cat = concatenate，顯示檔案內容到終端機。用來確認你的 inbox 裡有什麼，或確認剛才的 echo 有沒有成功存進去。
grep -r -l "關鍵字" ~/vault --include="*.md" 搜尋所有筆記裡包含某個關鍵字的檔案
grep = 搜尋文字內容。-r = recursive，包含所有子資料夾。-l = 只列出檔案名稱（不顯示每一行）。--include="*.md" = 只搜尋 .md 檔案。把「關鍵字」換成你要找的詞。
find ~/vault -name "*.md" | head -30 列出 Vault 裡的前 30 個 .md 檔案
find 找出所有符合條件的檔案。-name "*.md" = 名稱以 .md 結尾的檔案。| head -30 = 只顯示前 30 個（避免輸出太多）。用來快速查看 vault 的結構。

給 Claude 的指令範本

這些是你在 Claude Code 裡輸入的自然語言指令。

情境指令範本
開始工作 "讀取 projects/my-project/status.md，告訴我現在的狀態，等我說要做什麼"

08 — 進階技巧

讓工作流更強大的進階設定

這些不是必須的，但設定好之後能大幅提升效率。

技巧一：建立 Shell Alias，一個字啟動

每次都要輸入 cd ~/vault && claude 有點麻煩。設定一個 alias，之後只要打兩個字。

bash 設定 alias（只需要做一次） 
# 打開你的 shell 設定檔
# 如果你用 zsh（macOS 預設）：
nano ~/.zshrc

# 如果你用 bash：
nano ~/.bashrc

# 在檔案最後面加入這幾行：
alias vault="cd ~/vault && claude"
alias note="echo \"$(date +%Y-%m-%d) $*\" >> ~/vault/inbox/quick-capture.md"

# 儲存並重新載入（Ctrl+X, Y, Enter 存檔後）
source ~/.zshrc
# 或
source ~/.bashrc

設定好之後：

bash 使用 alias 
# 輸入 vault 直接啟動
vault

# 輸入 note 快速記錄
note "登入頁考慮加 Google OAuth"
# 會自動加上日期：2026-03-26 登入頁考慮加 Google OAuth

技巧二：用 iCloud 免費跨裝置同步

如果你有 iPhone 或 iPad，可以讓手機也能看到 Vault 的筆記。不需要付費的 Obsidian Sync。

bash 把 Vault 移到 iCloud（macOS） 
# iCloud 的 Obsidian 資料夾路徑
ICLOUD_PATH="~/Library/Mobile Documents/iCloud~md~obsidian/Documents"

# 建立 iCloud 資料夾（如果不存在）
mkdir -p "$ICLOUD_PATH"

# 把現有的 vault 移過去
mv ~/vault "$ICLOUD_PATH/vault"

# 建立 symlink，讓你還是可以用 ~/vault 這個路徑
# ln -s 原始路徑 連結名稱
ln -s "$ICLOUD_PATH/vault" ~/vault

# 確認 symlink 建立成功
ls -la ~ | grep vault
ℹ️

Symlink 的意思是「捷徑」。你之後還是可以用 ~/vault 這個路徑，但實際上資料存在 iCloud 資料夾裡，會自動同步到你的 iPhone。在手機上安裝 Obsidian app，選擇 iCloud 的 vault，就能讀寫。

技巧三：建立筆記範本，保持格式一致

每次記錄決定、bug、技術筆記，都要想「格式要怎麼寫」很煩。先建好範本，之後叫 Claude 照範本填。

bash 建立範本資料夾和範本檔案 
mkdir -p ~/vault/resources/templates

# 建立決策紀錄範本
cat > ~/vault/resources/templates/decision.md << 'EOF'
## YYYY-MM-DD [決定名稱]

**最終決定：** 

**背景：** （為什麼需要做這個決定）

**評估的選項：**
| 選項 | 優點 | 缺點 |
|------|------|------|
| 選項A |  |  |
| 選項B |  |  |

**選擇的理由：** 

**風險與緩解方式：** 

**重新評估條件：** （什麼情況下應該重新考慮這個決定）
EOF

# 建立 bug 紀錄範本
cat > ~/vault/resources/templates/bug-report.md << 'EOF'
## YYYY-MM-DD [Bug 標題]

**症狀：** （看到什麼錯誤、什麼行為）

**重現步驟：**
1. 

**根本原因：** （為什麼會發生）

**解法：**
```
（放修復的代碼）
```

**預防措施：** （下次怎麼避免）
EOF

之後叫 Claude：

指令 
"讀取 resources/templates/decision.md 的格式，
然後用這個格式把我們剛才的討論整理成決策紀錄，
存到 projects/my-project/decisions.md。"

技巧四：用 Daily Note 追蹤每日工作

如果你想要更細粒度的紀錄，可以用每日筆記，記錄當天做了什麼、遇到什麼問題。

bash 建立今日工作日誌 
# 自動建立今天日期的筆記
cat > ~/vault/daily-notes/$(date +%Y-%m-%d).md << 'EOF'
# 工作日誌

## 今天的目標
- 

## 完成的事
- 

## 遇到的問題
- 

## 明天要做的
- 

## 隨手記
EOF
最重要的原則

不要追求完美的結構。先開始記，之後再優化。一個不完美但有在用的 Vault，遠比一個完美但沒在用的 Vault 有價值。從最小可行的結構開始：CLAUDE.md + 一個 projects/ 資料夾 + inbox/，這樣就夠了。其他的等你真的需要再加。
